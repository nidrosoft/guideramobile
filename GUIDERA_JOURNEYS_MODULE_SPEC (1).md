## 0. How to use this document in Cursor

This spec is the single source of truth for building the **Journeys** module. Build it in the phase order in **§18**. Each section is written so you can implement file-by-file without guessing conventions — the SQL, TypeScript interfaces, system prompts, and Edge Function skeletons are production-shaped, not pseudocode.

**Hard rules for this module:**
1. **It is a separate module.** Everything lives under `src/modules/journeys/**` and `supabase/functions/journeys-*`. The rest of the app touches it only through the public API in `src/modules/journeys/index.ts` (§2.3). No deep imports from outside the module.
2. **It must run without the rest of the app.** No hard dependency on trips, bookings, or VIE. It reads `profiles` for identity/entitlement and (optionally) links to the existing `groups` community tables — both behind thin adapters so the module degrades gracefully if they're absent.
3. **Content is AI-generated by default, human-curated over time.** Every guide and search result follows the existing AI Engine pattern: module system prompt → Claude → strict JSON → cache by composite key → human-review lifecycle. AI never invents providers, prices as false precision, or gives medical/legal advice.
4. **UI is already designed** (see the approved mockups). This doc defines data, logic, prompts, gating, and flows — not pixels. Where UI is referenced, it's to define the *contract* the components consume.
5. **Conventions:** `gen_random_uuid()` PKs; user FK is `profiles(id) ON DELETE CASCADE` (= `auth.users`); `TIMESTAMPTZ DEFAULT NOW()`; `idx_*` indexes (partial where useful); RLS on every table; `status` text columns with `CHECK`; flexible content in `JSONB`; arrays as `TEXT[]`.

## 1. Overview & Strategic Framing

### 1.1 What Journeys is

Journeys organizes travel by **why** someone is crossing a border, not **where**. Each "journey" (Medical & Cosmetic, Relocation, Digital Nomad, Wellness, Retirement, Fertility, …) is an entry point into a deep, consistent, country-by-country experience: a universal guide, real costs, the process, risks, aftercare, verified providers, a peer community, and a cross-journey toolkit.

It targets the large, high-anxiety, high-value, **structurally underserved** segment that mainstream OTAs (Expedia/Kayak/Booking) ignore.

### 1.2 Why a separate, growable module

- It has its **own data model, AI prompts, navigation, and (later) chat** — a vertical product inside the app.
- It must be able to **grow**: add a journey, a sub-hub, a toolkit tool, or an AI concierge chat without touching unrelated code.
- It has **two entry points** (Home section + Community) and a **premium surface** (Guidera Pro), so it needs clean public seams.

### 1.3 The moat (drives the architecture)

AI assistants already answer "what's it like to get a hair transplant in Turkey." So **static AI guides are table stakes**, not the moat. The defensible value — and what Pro charges for — is:
- **Verified provider directories** (independently checked, lead-gen not pay-to-play),
- **The cross-journey toolkit** (visa tracker, cost calculator, document checklists, timeline planner, insurance/banking, emergency kit),
- **Peer matching + community** (talk to someone who's done this exact journey),
- **Personalization** (the module knows the user's saved journeys, country, and — optionally — their trips).

So: **breadth of journeys = top-of-funnel/SEO/word-of-mouth (cheap to stand up via AI generation); depth + monetization concentrate in providers, toolkit, and community.**

### 1.4 Monetization model per journey (important, encoded in the schema)

`journey_categories.monetization_model` ∈ `{affiliate, lead_gen, none}`:
- **affiliate** — lifestyle verticals (Nomad, Wellness, Relocation logistics, Retirement, Study, Adventure, Family, Heritage): standard affiliate/referral (insurance, banking/FX, coworking, retreats, tours, moving services).
- **lead_gen** — medical verticals (Medical & Cosmetic incl. Dental/Hair/Cosmetic, Fertility): **flat listing/lead-gen fees, never %-commission.** Percentage referral kickbacks on medical care create joint-and-several liability exposure (US FTC/DOJ). Providers are verified, ranked by quality signals, never by who paid.
- **none** — sensitive/excluded verticals handled as information/community only.

### 1.5 Sensitive / excluded verticals

`journey_categories.is_sensitive = true` + `monetization_model = none` for any vertical that is information/community only and **never** affiliate-monetized: gender-affirming care, addiction/rehab, anything asylum/refugee-adjacent. These are **not** seeded at launch and, if ever added, ship with legal review, no provider directory, no lead-gen, and reinforced disclaimers. The launch catalog (§19) excludes them entirely.

### 2.2 Dependency rules

- The module imports from `@/lib/supabase`, `@/lib/queryClient` (React Query), and `@/modules/journeys/**` only.
- Cross-feature touchpoints go through **adapters** inside the module:
  - `entitlements.adapter.ts` → reads `profiles.subscription_tier` (or your subscription source). If unavailable, defaults to free.
  - `journeyCommunity.service.ts` → reads/writes the existing `groups`/`group_members` tables **via the `journey_group_links` mapping only**. If those tables don't exist yet, community blocks render a "coming soon" state.
- Nothing outside the module may import its internal files; outside code uses `index.ts`.

## 3. Information Architecture, Entry Points & Navigation

### 3.1 Entry points

**A) Home page — "Journeys" section (`JourneysHomeSection`)**
- A titled section on the Home screen: heading **"Journeys"**, subtitle "Travel for a reason."
- A **horizontally scrollable rail** of journey cards (Medical & Cosmetic, Relocation, Digital Nomad, … up to `homeSection.maxCards`, ordered by `is_popular DESC, sort_order ASC`).
- A **"See all"** button (top-right of the section header) and a trailing "View all journeys →" card at the end of the rail.
- Tapping a journey card → `JourneysHubScreen` with that journey **pre-selected**.
- Tapping "See all" → `JourneysHubScreen` (default selection = first popular journey, e.g., Medical).
- Contract: the host renders `<JourneysHomeSection onSeeAll={...} />`; the component fetches the catalog itself (cached). The host passes nothing except optional style.

**B) Community / Connect tab — "Journeys" entry**
- Inside the Community tab (alongside Discover / My Groups / Buddies / Events), a **"Journeys"** entry (card or list row) → `JourneysHubScreen`.
- Rationale: communities/peer-matching are core to Journeys, so it belongs in Community too. Same destination screen, different doorway.

Both doorways land on the **same** `JourneysHubScreen`. No duplicated logic.

### 3.2 Screen map & flow

### 3.4 The Hub screen — exact behavior (matches approved UI)

Top → bottom on `JourneysHubScreen`:
1. **Header**: "Journeys / Travel for a reason. Find your people, places & playbook." + **search box** (placeholder rotates: `"Try \"hair transplant\" or \"retire cheap\""`).
2. **Pro Toolkit promo** (dark/gold) → `ToolkitScreen`.
3. **"Pick a journey" rail**: all live journeys, horizontally scrollable; selected card is filled with `tint`. Selecting sets `filter.categorySlug` and resets `filter.subhub` to `all`.
4. **Continent chips**: `All · Europe · Asia · Africa · Americas · Oceania`. Selecting sets `filter.continent`.
5. **Sub-hub chips** (only when selected journey `has_subhubs`, i.e., Medical): `Dental · Hair · Cosmetic`. Toggling sets `filter.subhub`.
6. **Result header**: `<icon> <Journey name> [in <Continent>] · N places`.
7. **Country list**: cards filtered by **journey × continent × sub-hub** (§7). Each card → `JourneyGuideScreen`.
8. **Empty state** (no matches for the combo) → inline note + the Generate card.
9. **"Don't see your country?" Generate card** → opens `GenerateGuideSheet` (§9.3) for the *current* journey + a country the user types/picks.
10. **Free vs Pro explainer** card.

Filter state is **client-side** over a cached catalog of guide *stubs* (not full guides) for snappy interaction (§7.2). Selecting a country fetches/generates the full guide (§8).

### 4.2 `journey_categories` — the journeys (16 at launch)

```sql
CREATE TABLE journey_categories (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    slug            TEXT UNIQUE NOT NULL,                 -- 'medical', 'relocation', ...
    name            TEXT NOT NULL,                        -- 'Medical & Cosmetic'
    subtitle        TEXT,                                 -- 'Treatment abroad'
    "group"         journey_group NOT NULL,               -- health | living | purpose
    icon            TEXT NOT NULL,                        -- icon key (see config/icons.ts), e.g. 'stethoscope'
    tint            TEXT NOT NULL,                        -- hex, e.g. '#C75D58'

    -- catalog behavior
    is_popular      BOOLEAN DEFAULT FALSE,                -- shows in Home rail + "Popular" badge
    has_subhubs     BOOLEAN DEFAULT FALSE,                -- Medical = true
    sort_order      INTEGER DEFAULT 100,

    -- strategy / compliance
    monetization_model  journey_monetization NOT NULL DEFAULT 'affiliate',
    risk_tier           journey_risk_tier NOT NULL DEFAULT 'low',
    is_sensitive        BOOLEAN DEFAULT FALSE,            -- info/community only; never seeded if true
    requires_disclaimer BOOLEAN DEFAULT FALSE,            -- medical/fertility/cbi/longevity

    -- AI generation guidance (injected into the guide system prompt; see §9.4)
    ai_definition       TEXT NOT NULL,                    -- 1-2 sentence scope of this journey
    ai_emphasis         TEXT NOT NULL,                    -- what to lead with / tone
    ai_section_order    TEXT[] NOT NULL,                  -- ordered section types (see §6)
    ai_critical_sections TEXT[] DEFAULT '{}',             -- which sections are "must populate well"
    ai_extra_fields     JSONB DEFAULT '{}',               -- vertical-specific quick-fact fields to include

    status          TEXT DEFAULT 'active' CHECK (status IN ('active','hidden','archived')),
    created_at      TIMESTAMPTZ DEFAULT NOW(),
    updated_at      TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_journey_categories_active ON journey_categories(status) WHERE status = 'active';
CREATE INDEX idx_journey_categories_group ON journey_categories("group");
CREATE INDEX idx_journey_categories_popular ON journey_categories(is_popular) WHERE is_popular = TRUE;
```

### 4.5 `journey_guides` — the universal guide (core table)

The `content` JSONB holds the full rendered guide (schema in §6). Stub fields (`hook`, `fit_tags`, `rating`, `cost_band`) are denormalized so the **catalog/filter** can render country cards without loading the full JSONB.

```sql
CREATE TABLE journey_guides (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    category_id     UUID NOT NULL REFERENCES journey_categories(id) ON DELETE CASCADE,
    subhub_id       UUID REFERENCES journey_subhubs(id) ON DELETE SET NULL,  -- nullable
    country_code    CHAR(2) NOT NULL REFERENCES journey_countries(code),
    focus           TEXT,                                 -- e.g. 'Hair Restoration' (display focus)

    -- composite cache key: category + country + subhub (empty string when null)
    cache_key       TEXT GENERATED ALWAYS AS (
                      category_id::text || ':' || country_code || ':' || COALESCE(subhub_id::text, '_')
                    ) STORED,

    -- lifecycle
    status          journey_guide_status NOT NULL DEFAULT 'ai_generated',
    source          journey_guide_source NOT NULL DEFAULT 'ai',
    is_published    BOOLEAN DEFAULT TRUE,

    -- denormalized stub (for country cards in the hub)
    hook            TEXT,
    fit_tags        TEXT[] DEFAULT '{}',                  -- ['FUE','DHI','Beard']
    headline_tag    TEXT,                                 -- 'Hair · Dental · Cosmetic'
    rating          NUMERIC(2,1),                         -- editorial/agg rating (display only)
    cost_band       TEXT,                                 -- '$','$$','$$$'

    -- full content
    content         JSONB NOT NULL DEFAULT '{}'::jsonb,   -- §6 GuideContent

    -- AI provenance
    model           TEXT,
    prompt_version  INTEGER,
    confidence      NUMERIC(3,2),                         -- 0.00–1.00 self-reported by model
    generated_at    TIMESTAMPTZ,

    -- human review
    reviewed_by     UUID REFERENCES profiles(id),
    reviewed_at     TIMESTAMPTZ,
    review_notes    TEXT,

    -- engagement (denormalized)
    view_count      INTEGER DEFAULT 0,
    save_count      INTEGER DEFAULT 0,

    created_at      TIMESTAMPTZ DEFAULT NOW(),
    updated_at      TIMESTAMPTZ DEFAULT NOW(),

    UNIQUE(cache_key)
);

CREATE INDEX idx_journey_guides_category ON journey_guides(category_id);
CREATE INDEX idx_journey_guides_country ON journey_guides(country_code);
CREATE INDEX idx_journey_guides_status ON journey_guides(status);
CREATE INDEX idx_journey_guides_published ON journey_guides(is_published) WHERE is_published = TRUE;
-- catalog/filter index: list guides for a journey, newest curated first
CREATE INDEX idx_journey_guides_catalog ON journey_guides(category_id, country_code) WHERE is_published = TRUE;
```

> **"Curated vs AI" badge** is derived: `status = 'curated'` → CURATED; `status IN ('ai_generated','pending_review')` → AI.

## 6. Section Model & Per-Vertical Configuration

The guide is a **list of typed sections** rendered top-to-bottom in an order that **adapts per journey**. This is what makes "Medical leads with cost/process/risk/aftercare" while "Solo Female leads with safety/community" — same renderer, different order + emphasis, driven by `journey_categories.ai_section_order`.

### 6.1 Section types (`SectionType`)

```ts
export type SectionType =
  | "quick_facts"      // 4-up grid (vertical-aware fields)
  | "things_to_know"   // UNIVERSAL bullets
  | "why_here"         // category-flavored prose
  | "costs"            // UNIVERSAL comparison table (vs home country)
  | "process"          // numbered steps
  | "logistics"        // getting there/around, housing, banking, connectivity
  | "top_destinations" // cities/regions -> link to Destination Snapshots
  | "risks"            // red flags & scams
  | "aftercare"        // recovery/follow-up (medical/dental/fertility)  [NEW]
  | "legal"            // what's permitted (fertility/cbi/medical)
  | "providers"        // PRO — verified directory (rendered from journey_providers, NOT AI)
  | "community"        // group link + peer matching
  | "faq";             // generated Q&A
// 'hero' + 'disclaimer' are rendered by the screen chrome, not as body sections.
```

### 6.2 GuideContent JSONB schema (what the model returns + what the renderer consumes)

## 7. Filtering Logic (journey × continent × sub-hub)

### 7.1 What the hub filters over

The hub filters over **`GuideStub[]`** — lightweight rows, NOT full guides. A stub exists for any (category × country [× subhub]) that has a published guide. For breadth, every journey also has a small **seed set** of "known destination" stubs even before a full guide exists (status `'none'` → tapping triggers generate-on-demand).

### 7.2 Catalog fetch (cached, snappy)

```ts
// journeyContent.service.ts
export async function getCatalogStubs(categorySlug: string): Promise<GuideStub[]> {
  // 1) published guides for this journey (curated first)
  // 2) UNION seeded "known destinations" (from a static map per journey) lacking a guide => status 'none'
  // returns merged, de-duped by countryCode (+subhub), sorted: curated > ai > none, then rating desc
}
```

Cache with React Query: `["journeys","catalog",categorySlug]`, `staleTime: 5min`.

### 7.3 Client filter (`hooks/useJourneyFilter.ts`)

```ts
export function useJourneyFilter() {
  const { filter, setCategory, setContinent, setSubhub } = useJourneyFilterStore();
  const { data: stubs = [] } = useCatalogStubs(filter.categorySlug);

  const results = useMemo(() => stubs.filter(s =>
    (filter.continent === "All" || s.continent === filter.continent) &&
    (filter.subhubSlug === "all" || !s.subhubSlugs || s.subhubSlugs.includes(filter.subhubSlug))
  ), [stubs, filter]);

  return { filter, results, setCategory, setContinent, setSubhub };
}
```

Rules:
- **Selecting a journey** resets `subhubSlug` to `"all"` (sub-hubs are journey-specific) and keeps `continent`.
- **Sub-hub chips** render only when the selected category `hasSubhubs`.
- **Empty result** (e.g., `medical × Africa`) → empty-state note + Generate card (the model can produce a guide for any country, so nothing is a dead end).
- Result header text: `"<Journey> in <Continent> · N places"` (omit "in <Continent>" when `All`).

### 7.4 Country card → guide open

Tapping a country card:
- If `stub.guideId` exists → `openGuide(categorySlug, countryCode, subhubSlug?)` → `JourneyGuideScreen` loads it.
- If `status === 'none'` → still navigate to `JourneyGuideScreen`, which shows a **generating** state and calls generate-or-fetch (§8.2) — i.e., the country *behaves* the same whether curated or generated; the only difference is the badge and a "verify details" note.

## 8. Content Sources: Curated vs AI-Generated

### 8.1 One object, two fill methods

A guide row is the **same** whether hand-curated or AI-generated; `status` + `source` distinguish them. This means one renderer, one schema, two pipelines, and a smooth promotion path (`ai_generated → pending_review → curated`).

### 8.2 Generate-or-fetch (the core content flow)

```ts
// journeyContent.service.ts
export async function getOrGenerateGuide(params: {
  categorySlug: string; countryCode: string; subhubSlug?: string; userId?: string;
}): Promise<JourneyGuide> {
  // 1) Look up by cache_key. If found AND fresh (< guideStaleDays) AND published => return.
  // 2) If missing or stale => call Edge Function `journeys-generate-guide` (service-role writes the row).
  // 3) Return the (new or refreshed) guide. Mark status 'ai_generated' unless curated.
}
```

`JourneyGuideScreen` calls this via `useJourneyGuide({categorySlug, countryCode, subhubSlug})`:
- Shows skeleton + `GenerateGuideSheet` progress when generating.
- On success, renders sections per `order`, injects providers/community, applies Pro gates.

### 8.3 Lifecycle & human review

```
ai_generated ──(editor opens in admin)──► pending_review ──(approve)──► curated
        │                                                          │
        └────────────── feedback flags / staleness ───────────────┘ (re-review / regenerate)
```

- New AI guides are immediately visible (badge **AI**, with `generatedNote = "AI-generated — verify details"`).
- An internal admin queue lists `status IN ('ai_generated','pending_review')` ordered by `view_count DESC` (review what users actually see).
- Approving sets `status='curated'`, `source='hybrid'` if edited, clears the note, shows **CURATED** badge.
- `journey_guide_feedback` with `flag_reason IN ('inaccurate','outdated','unsafe')` re-opens review.

### 8.4 Cost controls (matches existing AI Engine economics)

- **Anthropic prompt caching**: the large, static parts of the guide system prompt + the per-journey spec block are sent with `cache_control` so repeated generations across countries hit the ~90% cached discount.
- **Batch API** for the nightly enrichment job (`journeys-enrich-batch`) that pre-generates guides for `popular journey × top countries` and refreshes stale ones (~50% cheaper, non-urgent).
- **Staleness, not TTL spam**: guides regenerate only when older than `guideStaleDays` or flagged.

## 9. AI System Prompts (the heart of the module)

Two prompts power Journeys, both following the existing AI Engine contract: **strict JSON only, no prose, validate with zod, cache by key**. Both run server-side in Edge Functions with the service-role key.

> **Global guardrails (apply to BOTH prompts):**
> - Output **only** what is genuinely true/known. Never invent statistics, clinics, success rates, or laws. Prefer **ranges** over false precision.
> - **No medical, legal, financial, or immigration advice.** Information only. For medical/fertility/CBI/longevity verticals, frame as general info and assume the screen will show a disclaimer.
> - **Never output specific named providers/clinics/agencies** — those come from the verified `journey_providers` table, not the model.
> - If a fact is uncertain or varies by nationality/region, **say so** and give a conservative range.
> - Be specific and concrete (countries, corridors, real costs), not generic travel-brochure language.

### 9.1 Model + caching wrapper (`functions/_shared/claude.ts`)

```ts
import Anthropic from "npm:@anthropic-ai/sdk";
const anthropic = new Anthropic({ apiKey: Deno.env.get("ANTHROPIC_API_KEY")! });

export async function callClaudeJSON(opts: {
  model: string; maxTokens: number;
  system: { staticPrompt: string; dynamicSpec?: string };  // staticPrompt is cached
  userContent: string;
}) {
  const res = await anthropic.messages.create({
    model: opts.model,
    max_tokens: opts.maxTokens,
    system: [
      // cached block — the big invariant instructions
      { type: "text", text: opts.system.staticPrompt, cache_control: { type: "ephemeral" } },
      // per-call spec (journey definition etc.) — cached too if reused across countries in a batch
      ...(opts.system.dynamicSpec ? [{ type: "text", text: opts.system.dynamicSpec, cache_control: { type: "ephemeral" } }] : []),
    ],
    messages: [{ role: "user", content: opts.userContent }],
  });
  const text = res.content.find(b => b.type === "text")?.text ?? "";
  return text; // caller extracts + validates JSON (functions/_shared/json.ts)
}
```

`functions/_shared/json.ts` strips ```json fences, `JSON.parse`, then validates against the zod schema; on failure it retries once with a "RETURN VALID JSON ONLY" nudge, then throws.

### 9.2 SEARCH prompt — "What is `<country>` known for, across journeys?"

**Purpose:** When a user types a free-text query into the hub search box, resolve it and — for a country query — produce an honest profile of **what that country is genuinely known for**, mapped only to the journeys where it actually has a reputation. This powers `JourneySearchResultsScreen`.

**Query resolution (before the prompt) — `journeySearch.service.ts`:**
1. Normalize the query. Try to resolve a **country** (fuzzy match against `journey_countries.name` + common aliases) and/or a **journey intent** (keyword map → category slug, e.g. "hair transplant"→medical/hair, "retire cheap"→retire, "IVF"→fertility, "digital nomad"→nomad).
2. Routing:
   - **country + category** → go straight to `getOrGenerateGuide` → open the guide.
   - **category only** → open the hub pre-filtered to that journey.
   - **country only (or country + vague intent)** → fetch/generate the **country profile** (this prompt) → `JourneySearchResultsScreen`.
   - **nothing resolvable** → show suggestions (popular journeys) + free-form "generate" affordance.
3. Cache: country profiles live in `journey_country_profiles` (refresh after `searchProfileStaleDays`). Log every search to `journey_search_queries`.

**System prompt (`ai/prompts/search.prompt.ts`):**

```

country has a real, recognized reputation.

THE FIXED SET OF JOURNEYS (use these exact slugs):
medical, relocation, nomad, wellness, retire, fertility, solo, study, pilgrimage, adventure,
heritage, longevity, cbi, worldschool, volunteer, family.

WHAT EACH JOURNEY MEANS (match only if the country is genuinely notable for it):
- medical: surgery, dental, hair restoration, cosmetic, bariatric — affordable/quality care abroad
- relocation: long-term moving / expat life (visas, cost of living, housing)
- nomad: remote work bases (visa, internet, community, cost)
- wellness: retreats, yoga, detox, spiritual reset
- retire: retiring abroad on a fixed income (healthcare, cost, climate)
- fertility: IVF / egg donation / surrogacy (legal framework matters)
- solo: solo female travel safety
- study: universities / semesters abroad
- pilgrimage: faith/spiritual sites and routes
- adventure: trekking, climbing, expeditions, nature
- heritage: ancestry / roots / diaspora reconnection

3. For each matched journey: a punchy 4–8 word "headline" and a 1-sentence "why" grounded in fact

### 9.3 GUIDE GENERATION prompt — full universal guide for `<journey × country>`

**Purpose:** Produce the complete `GuideContent` (§6.2) for a specific journey + country (+ optional sub-hub), in that journey's section order, with vertical-appropriate emphasis.

**System prompt (`ai/prompts/guide.prompt.ts`) — STATIC, cached block:**

```
You are Guidera's Journey Guide engine. You write a single, trustworthy, country-specific guide
for ONE purposeful travel journey. A first-timer should finish your guide feeling as prepared as
someone who already did it. Output strict JSON only.

ABSOLUTE RULES:
- Output ONLY valid JSON matching the provided schema. No markdown, no commentary outside JSON.
- Truthful and specific. Use RANGES for costs/durations; never invent precise figures, success
  rates, or laws. If something varies by nationality/region/clinic, say so.
- NO advice (medical/legal/financial/immigration). Information framing only.
- DO NOT list specific named providers, clinics, agencies, hospitals, or firms anywhere. Provider
  directories are handled separately by Guidera. Refer to provider TYPES generically.
- DO NOT include "providers" or "community" sections — Guidera injects those from verified data.
- Universal sections ("things_to_know", "costs") must keep their universal intent and structure.
- "costs" must compare to the traveler's likely home country generically (e.g., "vs US/UK").
- "top_destinations" lists 2–4 real cities/regions with a one-line reason each.
- Be honest about RISKS and red flags — do not soften. This is a safety feature.
- For aftercare-relevant journeys (medical/fertility), give realistic recovery/follow-up guidance
  and clear "don't do X yet" timing (e.g., flying after a procedure).
- Set "confidence" (0–1) reflecting how well-established the facts are for this journey×country.
- "requiresDisclaimer" must be true for medical, fertility, cbi, longevity.

You will receive a JOURNEY SPEC (definition, emphasis, the exact ordered list of section types to
produce, which sections are critical, and vertical-specific quick-fact fields) and the COUNTRY.
Produce sections in EXACTLY the given order, OMITTING any "providers"/"community" types if present
in the order (Guidera injects them at that position). Populate "critical" sections most richly.
```

**DYNAMIC spec block (also cached within a batch) — built from `journey_categories` + `SECTION_CONFIG`:**

```
JOURNEY: {name} (slug: {slug})
DEFINITION: {ai_definition}
EMPHASIS: {ai_emphasis}
{If subhub}: FOCUS: {subhub.name} — {subhub.ai_focus}
SECTION ORDER (produce these, in this order; skip providers/community): {order joined}
CRITICAL SECTIONS (populate richly): {critical joined}
QUICK-FACT FIELDS TO INCLUDE: {quickFactFields joined}
RISK POSTURE: {high|medium|low} → {if high: be especially explicit about safety/accreditation/scams}
DISCLAIMER REQUIRED: {true/false}
```

**User content:** `COUNTRY: Turkey (TR, Asia). Generate the guide as specified.`

**Output JSON schema** = `GuideContent` (§6.2). The Edge Function validates with `ai/schemas/guide.schema.ts` (zod), strips provider/community types if the model included them, and writes the row (denormalizing `hook`, `fit_tags`, `headline_tag`, `cost_band`).

### 9.4 Why this design is the moat-aware version

The guide JSON is deliberately **generatable by any LLM** — that's fine, because Guidera's defensibility is the *system around it*: the verified provider directory, the toolkit, the community/peer-matching, the curation/quality loop, and the fact that it's embedded in a trip-aware app. The prompts intentionally **refuse** to emit the high-trust, high-liability pieces (named providers, precise medical claims) — those are owned by Guidera's data and humans.

## 10. Edge Functions (Deno + `@anthropic-ai/sdk`)

### 10.1 `journeys-generate-guide`

```ts
// supabase/functions/journeys-generate-guide/index.ts
import { createClient } from "npm:@supabase/supabase-js";
import { callClaudeJSON } from "../_shared/claude.ts";
import { extractJSON, validateGuide } from "../_shared/json.ts";
import { buildGuideSpec, GUIDE_SYSTEM_PROMPT } from "./prompt.ts";
import { corsHeaders } from "../_shared/cors.ts";

Deno.serve(async (req) => {
  if (req.method === "OPTIONS") return new Response("ok", { headers: corsHeaders });
  const { categorySlug, countryCode, subhubSlug } = await req.json();

  const admin = createClient(Deno.env.get("SUPABASE_URL")!, Deno.env.get("SUPABASE_SERVICE_ROLE_KEY")!);

  // load category (+subhub) + country + section config
  const { data: cat } = await admin.from("journey_categories").select("*").eq("slug", categorySlug).single();
  if (!cat || cat.is_sensitive) return json(400, { error: "invalid_or_sensitive_category" });
  const { data: country } = await admin.from("journey_countries").select("*").eq("code", countryCode).single();
  const subhub = subhubSlug
    ? (await admin.from("journey_subhubs").select("*").eq("category_id", cat.id).eq("slug", subhubSlug).single()).data
    : null;

  // build cached spec block
  const dynamicSpec = buildGuideSpec(cat, subhub);

  // generate
  const raw = await callClaudeJSON({
    model: Deno.env.get("JOURNEYS_CLAUDE_MODEL") ?? "claude-sonnet-4-20250514",
    maxTokens: 4096,
    system: { staticPrompt: GUIDE_SYSTEM_PROMPT, dynamicSpec },
    userContent: `COUNTRY: ${country.name} (${country.code}, ${country.continent}). Generate the guide as specified.`,
  });
  const content = validateGuide(extractJSON(raw)); // zod; strips providers/community types

  // upsert by cache_key (DB computes cache_key)
  const { data: guide } = await admin.from("journey_guides").upsert({
    category_id: cat.id, subhub_id: subhub?.id ?? null, country_code: country.code,
    focus: content.hero.focus ?? subhub?.name ?? null,
    status: "ai_generated", source: "ai", is_published: true,
    hook: content.hero.hook, fit_tags: content.hero.fitTags,
    headline_tag: deriveHeadlineTag(content), cost_band: deriveCostBand(content),
    content, model: Deno.env.get("JOURNEYS_CLAUDE_MODEL"),
    prompt_version: 3, confidence: content.confidence, generated_at: new Date().toISOString(),
  }, { onConflict: "cache_key" }).select().single();

  // emit event
  await admin.from("journey_events").insert({ event_type: "guide_generated", category_slug: categorySlug, country_code: countryCode, payload: { confidence: content.confidence } });

  return json(200, { guide });
});
```

## 13. Verified Providers (Pro) + lead-gen model

- Rendered by `ProviderList.tsx` from `journey_providers` for the guide's (category × country [× subhub]); **ranked by `quality_score` and `is_verified`, never by payment.**
- Each provider row: name, type, accreditation chips (JCI/TEMOS/ISO), verified badge, summary. Tap → `ProviderDetailSheet` (bottom sheet) with full detail + a **"Request info"** action that writes `journey_provider_leads` (lead-gen) and opens the provider's contact channel.
- **Monetization:** medical providers are `monetization = 'lead_gen'` (flat fee per qualified lead/listing). No %-commission on medical care. Lifestyle verticals may use affiliate links via the same UI but flagged `affiliate`.
- **Compliance:** provider listing ≠ endorsement. The guide screen always renders the medical/legal disclaimer when `requiresDisclaimer`. Verification notes are stored (`verification_notes`) for auditability.

### 16.3 Bottom sheets (interaction contracts)

- **`GenerateGuideSheet`** — opens when generating on demand. Shows journey + country, a progress state (calling AI), then either routes into the new guide or shows an error/retry. Cancelable. Triggered from the hub Generate card and from `JourneyGuideScreen` cold-load.
- **`ProviderDetailSheet`** — provider detail; "Request info" → `captureLead` → opens contact channel; shows accreditation + verification note + disclaimer.
- **`ProUpsellSheet`** — paywall; reads global pricing; CTA to subscribe; dismiss returns to the locked preview.
- **Filter behavior is inline (not a sheet):** chips/rail update the list live (§7). Only generation, provider detail, and paywall use sheets.

## 18. Implementation Sequencing (build order)

**Phase 2 — AI generation, search, providers, toolkit core (Week 3–5)**

8. Toolkit: cost calculator + checklist (the two that demo the "tools > guides" thesis). Pro gating wired to real entitlement.

**Acceptance:** searching "Cameroon" yields an honest cross-journey profile; tapping an unseeded country generates a guide; Pro users see providers + toolkit; free users see locked previews.

**Phase 3 — Growth surfaces (Week 6+)**
9. `journeys-enrich-batch` cron (pre-generate popular journey × top countries; refresh stale). 
10. Peer matching matcher + notifications. Remaining toolkit tools (visa alerts, timeline, insurance/banking, emergency).
11. **AI concierge chat** (enable `chat`), per-journey scoped prompt; optional tool-calls into toolkit.
12. Curation admin queue + feedback-driven re-review.
